# Background commands

Use the **Commands API** when a workload should **outlive a single HTTP request**: dev servers, queue workers, install scripts that print for minutes, or anything you need to **observe** or **stop** later by **process id (PID)**.

Blocking **`POST .../exec`** waits until the command finishes (or hits a timeout). **`exec-stream`** streams output but still ties the client to that request. Background commands **return immediately** with a **PID** so you can **list**, **attach** (SSE), **wait**, or **kill** from separate calls.

<Note>
For a full comparison with sync **`exec`**, streaming, PTY, and **`runCode`**, see [Shell execution](/core-concepts/shell-execution). Detached processes with a **PID** are started only via **`POST .../commands/run`** (and **`sandbox.commands.run`** in SDKs); use **list / wait / kill / attach** from the Commands API.
</Note>

## When to use what

| Approach | Best for |
|----------|----------|
| **Sync `exec`** | Short commands; you want **stdout/stderr/exitCode** in one response. |
| **`exec-stream`** | Long output, but you are fine **holding one streaming request** until exit. |
| **Commands API (`/commands/run`, â€¦)** | **Canonical** model for **PID lifecycle**: run, list, attach logs, wait for exit, kill. |

## REST endpoints

All routes are under **`/sandboxes/{id}/commands/`** (prefix with your API base, e.g. **`https://platform.void-run.com/api`**). Authenticate with **`X-API-Key`**.

| Method | Path | Role |
|--------|------|------|
| **POST** | `.../commands/run` | Start a background process; body **`command`** (required), optional **`env`**, **`cwd`**, **`timeout`**. |
| **GET** | `.../commands/list` | List tracked processes and metadata. |
| **POST** | `.../commands/attach` | **SSE** stream of **`stdout`**, **`stderr`**, **`exit`** for a **PID** (JSON body **`{ "pid": number }`**). |
| **POST** | `.../commands/wait` | Block until the **PID** exits; returns **exitCode**. |
| **POST** | `.../commands/kill` | Terminate a **PID**. |

Request/response schemas are in the **API Reference** tab (e.g. **`runBackgroundCommand`**, **`attachToBackgroundProcess`**, **`waitForBackgroundProcess`**).

### Run request (`/commands/run`)

- **`command`** (string, required): shell command line as executed in the sandbox.
- **`env`** (object, optional): extra environment variables.
- **`cwd`** (string, optional): working directory.
- **`timeout`** (integer, optional): seconds. Per OpenAPI, **`0`** means **no timeout**; maximum **900** when a limit applies.

### Responses you will see

- **Run:** **`success`**, **`pid`**, **`command`** (`CommandRunResponse`).
- **List:** **`success`**, **`processes`** â€” each item can include **`pid`**, **`command`**, **`startTime`**, **`running`**, **`exitCode`** (`ProcessInfo`).
- **Wait:** **`success`**, **`exitCode`**, optional **`error`** (`CommandWaitResponse`).
- **Kill:** **`success`**, **`message`** (`CommandKillResponse`).
- **Attach:** **`text/event-stream`** with events **`stdout`**, **`stderr`**, **`exit`** (and **`error`** when applicable).

## Typical flows

1. **Fire and reap:** **`run`** â†’ **`wait`** â†’ read **`exitCode`** (batch workers).
2. **Fire and stream:** **`run`** â†’ **`connect`** / attach (SSE) until **`exit`** (logs).
3. **Inspect:** **`list`** to debug PIDs and commands.
4. **Stop:** **`kill`** for stuck jobs or cancellation.

<Warning>
**`wait`** and **`attach`** both relate to the same PID: **`wait`** returns JSON when the process exits; **`attach`** consumes an **SSE** stream. For one PID, use **either** streaming **or** **`wait`** in the same way your SDK expectsâ€”avoid overlapping both unless you know your stack allows it.
</Warning>

## SDK notes

- **TypeScript:** **`sandbox.commands.run(command, env?, cwd?, timeout?)`** â€” **positional** arguments, not an options object. **`run`** returns **`pid`** on the **response object** (top level). **`list()`** returns **`{ processes }`**. **`wait`**, **`kill`**, **`connect`** take the numeric **PID**.
- **Python:** Same **positional** order **`(command, env, cwd, timeout)`**; use **`None`** for unused slots. Responses are often **`VoidRunResponse`**: use **`result.data.pid`**, **`result.data.exit_code`**, etc.

## Examples

Replace **`sandbox-id`** in SDK strings and **`<sandbox-id>`** in CLI/cURL with your sandbox id. Set **`VR_API_KEY`**. cURL uses the hosted API host; on self-hosted, swap the URL for your **`VR_API_URL`**.

### Start a background process (`commands/run`)

<CodeGroup>
```typescript TypeScript
import { VoidRun } from '@voidrun/sdk';

const vr = new VoidRun({ apiKey: process.env.VR_API_KEY! });
const sandbox = await vr.getSandbox('sandbox-id');

const run = await sandbox.commands.run(
  'npm run dev',
  { NODE_ENV: 'development' },
  '/app',
  0,
);

console.log('PID:', run.pid, 'command:', run.command);
```

```python Python
from voidrun import VoidRun
import os

vr = VoidRun(api_key=os.environ.get("VR_API_KEY"))
sandbox = vr.get_sandbox("sandbox-id")

run = sandbox.commands.run(
    "npm run dev",
    {"NODE_ENV": "development"},
    "/app",
    0,
)

print("PID:", run.data.pid, "command:", run.command)
```

```bash CLI
voidrun bg run <sandbox-id> --command 'npm run dev' --cwd /app
```

```bash cURL
curl -X POST "https://platform.void-run.com/api/sandboxes/<sandbox-id>/commands/run" \
  -H "X-API-Key: $VR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "command": "npm run dev",
    "cwd": "/app",
    "env": {"NODE_ENV": "development"},
    "timeout": 0
  }'
```
</CodeGroup>

### Wait until a process exits

Use the **PID** returned from **`run`**. Replace **`<pid>`** with that number in CLI and cURL.

<CodeGroup>
```typescript TypeScript
import { VoidRun } from '@voidrun/sdk';

const vr = new VoidRun({ apiKey: process.env.VR_API_KEY! });
const sandbox = await vr.getSandbox('sandbox-id');

const run = await sandbox.commands.run(
  'sleep 2 && echo done',
  undefined,
  '/tmp',
  0,
);

const done = await sandbox.commands.wait(run.pid);
console.log('Exit code:', done.exitCode);
```

```python Python
from voidrun import VoidRun
import os

vr = VoidRun(api_key=os.environ.get("VR_API_KEY"))
sandbox = vr.get_sandbox("sandbox-id")

run = sandbox.commands.run("sleep 2 && echo done", None, "/tmp", 0)
done = sandbox.commands.wait(run.data.pid)
print("Exit code:", done.data.exit_code)
```

```bash CLI
voidrun bg wait <sandbox-id> --pid <pid>
```

```bash cURL
# Replace 12345 with the PID from commands/run
curl -X POST "https://platform.void-run.com/api/sandboxes/<sandbox-id>/commands/wait" \
  -H "X-API-Key: $VR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"pid": 12345}'
```
</CodeGroup>

### Attach and stream output (SSE)

Stream **`stdout`** / **`stderr`** until an **`exit`** event. If the process exits very quickly, attach may see little outputâ€”**`wait`** is safer for short jobs.

<CodeGroup>
```typescript TypeScript
import { VoidRun } from '@voidrun/sdk';

const vr = new VoidRun({ apiKey: process.env.VR_API_KEY! });
const sandbox = await vr.getSandbox('sandbox-id');

const run = await sandbox.commands.run(
  'for i in 1 2 3; do echo "line $i"; sleep 1; done',
  undefined,
  '/tmp',
  0,
);

await sandbox.commands.connect(run.pid, {
  onStdout: (data) => process.stdout.write(data),
  onStderr: (data) => process.stderr.write(data),
  onExit: (payload) => console.log('exited', payload),
});
```

```python Python
import sys
from voidrun import VoidRun
import os

vr = VoidRun(api_key=os.environ.get("VR_API_KEY"))
sandbox = vr.get_sandbox("sandbox-id")

run = sandbox.commands.run(
    'for i in 1 2 3; do echo "line $i"; sleep 1; done',
    None,
    "/tmp",
    0,
)

sandbox.commands.connect(
    run.data.pid,
    on_stdout=lambda d: print(d, end=""),
    on_stderr=lambda d: print(d, end="", file=sys.stderr),
    on_exit=lambda p: print("exited", p),
)
```

```bash CLI
# Many CLI builds do not expose attach; use cURL (-N) or the SDK.
voidrun bg run <sandbox-id> --command 'echo use SDK or cURL for attach'
```

```bash cURL
# Replace 12345 with the PID from commands/run
curl -N -X POST "https://platform.void-run.com/api/sandboxes/<sandbox-id>/commands/attach" \
  -H "X-API-Key: $VR_API_KEY" \
  -H "Content-Type: application/json" \
  -H "Accept: text/event-stream" \
  -d '{"pid": 12345}'
```
</CodeGroup>

### Full lifecycle (run â†’ list â†’ wait â†’ kill)

Same pattern as [Code execution Â§ Background processes](/core-concepts/code-execution#background-processes): start a process, inspect the list, **either** wait for exit **or** kill (not both required).

<CodeGroup>
```typescript TypeScript
import { VoidRun } from '@voidrun/sdk';

const vr = new VoidRun({ apiKey: process.env.VR_API_KEY! });
const sandbox = await vr.getSandbox('sandbox-id');

const runResult = await sandbox.commands.run('npm run dev', undefined, '/app', 0);
console.log('Process PID:', runResult.pid);

const listResult = await sandbox.commands.list();
console.log('Running processes:', listResult.processes);

const waitResult = await sandbox.commands.wait(runResult.pid);
console.log('Exit code:', waitResult.exitCode);

// Or stop early:
// await sandbox.commands.kill(runResult.pid);
```

```python Python
from voidrun import VoidRun

vr = VoidRun(api_key=os.environ.get("VR_API_KEY"))

run_result = sandbox.commands.run("npm run dev", None, "/app", 0)

list_result = sandbox.commands.list()
print("Running processes:", list_result.data)

wait_result = sandbox.commands.wait(run_result.data.pid)
print("Exit code:", wait_result.data.exit_code)

# sandbox.commands.kill(run_result.data.pid)
```

## Practices

- **Sandbox must be running** â€” start or resume the VM before **`commands/run`**.
- **PID is scoped to the sandbox** â€” do not reuse PIDs across sandboxes.
- **Fast commands** â€” prefer **`wait`** for very short jobs; **`attach`** needs the process to still be producing output when you connect.
- **Teardown** â€” **`remove()`** the sandbox when finished; **`kill`** is for in-session control.
