# Custom Images

VoidRun runs each sandbox from a **catalog image** you select with the **`image`** field (and optional **`name:version`** tag). The hosted platform ships **three system images**: **`code`**, **`max`**, and **`docker`**: documented below. You still get **MicroVM isolation** and fast resume for sleeping sandboxes.

## System images

These are the **built-in system images** on the VoidRun platform (your dashboard lists them under **System images**). Version numbers change over time (for example `code:1.0.47`); you pass the **short name** when the API defaults the version, or **`name:version`** when you pin.

| Image | Base | What you use it for | Notes |
|-------|------|---------------------|--------|
| **`code`** | Alpine (thin) | Python, JavaScript, Node.js, TypeScript, interpreters, and most SDK examples | **Time to interact (TTI) &lt; ~300 ms** on the platform; ~5 GB image size. |
| **`max`** | Debian | General-purpose Linux, broader compatibility with Debian/Ubuntu-style tooling and scripts | **TTI &lt; ~2 s**; ~5 GB image size. |
| **`docker`** | Docker-in-Docker | Running Docker (and similar container workflows) **inside** the sandbox | Larger footprint (**~20 GB**); allocate enough **memory** and **disk** for the daemon and your containers. |

<Note>
You confirm exact names, versions, and sizes in the [dashboard](https://dash.void-run.com) or with **`GET /images`**. Self-hosted deployments may expose a different catalog: see [Self-hosted deployments](/advanced/self-hosted).
</Note>

If you **omit** **`image`**, the API uses the deployment default (the open-source SDKs default to **`code`** when they fill the field: see [TypeScript](/sdks/typescript) / [Python](/sdks/python) SDK pages).

## How images work

When you create a sandbox with a given image, VoidRun:

1. **Resolves** the image (from cache or pull, depending on the host)
2. **Boots an isolated MicroVM** with that root filesystem
3. **Keeps resume fast** when the sandbox sleeps (platform-dependent; often **under ~200 ms**)

## Example: `code` image (Node and Python)

The **`code`** image includes Node and Python tooling. You use it for JavaScript/TypeScript **and** Python workloads without switching images.

<CodeGroup>
```typescript TypeScript
import { VoidRun } from '@voidrun/sdk';

const vr = new VoidRun({ apiKey: process.env.VR_API_KEY! });

const sandbox = await vr.createSandbox({
  name: 'app-sandbox',
  image: 'code',
  cpu: 2,
  mem: 2048,
});

console.log('Sandbox ID:', sandbox.id);
console.log('Image:', sandbox.image);

const nodeVersion = await sandbox.exec({ command: 'node --version' });
console.log('Node:', nodeVersion.data?.stdout?.trim());

const pyVersion = await sandbox.exec({ command: 'python3 --version' });
console.log('Python:', pyVersion.data?.stdout?.trim());

const codeResult = await sandbox.runCode(
  'print(2 + 2)',
  { language: 'python' },
);
console.log(codeResult.stdout.trim());

await sandbox.remove();
```

```python Python
from voidrun import VoidRun

vr = VoidRun()

sandbox = vr.create_sandbox(
    name="app-sandbox",
    image="code",
    cpu=2,
    mem=2048,
)
print("Sandbox ID:", sandbox.id)
print("Image:", sandbox.image)

print("Node:", sandbox.exec("node --version").data.stdout.strip())
print("Python:", sandbox.exec("python3 --version").data.stdout.strip())
print(sandbox.run_code("print(2 + 2)", language="python").stdout.strip())

sandbox.remove()
```

```bash CLI
voidrun create --name app-sandbox --template code --cpu 2 --mem 2048
voidrun exec <sandbox-id> --command "node --version"
voidrun exec <sandbox-id> --command "python3 --version"
```

```bash cURL
curl -X POST "https://platform.void-run.com/api/sandboxes" \
  -H "X-API-Key: $VR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "app-sandbox",
    "image": "code",
    "cpu": 2,
    "mem": 2048
  }'
```
</CodeGroup>

## Example: `max` image (Debian general purpose)

You choose **`max`** when you need a **Debian-style** environment and broader compatibility than the Alpine-based **`code`** image.

<CodeGroup>
```typescript TypeScript
const sandbox = await vr.createSandbox({
  name: 'debian-style-job',
  image: 'max',
  cpu: 2,
  mem: 4096,
});

const result = await sandbox.exec({ command: 'cat /etc/os-release | head -n 3' });
console.log(result.data?.stdout);

await sandbox.remove();
```

```python Python
sandbox = vr.create_sandbox(name="debian-style-job", image="max", cpu=2, mem=4096)
print(sandbox.exec("cat /etc/os-release | head -n 3").data.stdout)
sandbox.remove()
```
</CodeGroup>

## Example: `docker` image (containers inside the sandbox)

You choose **`docker`** for **Docker-in-Docker** workflows. You need more **RAM** and **disk** than a typical **`code`** sandbox. See [Nested containers](/advanced/nested-containers) for full examples.

```typescript
const sandbox = await vr.createSandbox({
  name: 'dind-job',
  image: 'docker',
  cpu: 4,
  mem: 8192,
});
```

## Discovering images

The HTTP API lists available images with **`GET /images`**. Use the **`name`** (and **`version`** if you pin) returned there as **`image`** on **`POST /sandboxes`**.

<CodeGroup>
```typescript TypeScript
import { VoidRun } from '@voidrun/sdk';

const apiKey = process.env.VR_API_KEY!;

const vr = new VoidRun({ apiKey });
// Same default host as OpenAPI `servers` / cURL below; self-hosted: pass `baseUrl` to `VoidRun` and use that root here.
const apiRoot = 'https://platform.void-run.com/api';

const res = await fetch(`${apiRoot.replace(/\/$/, '')}/images`, {
  headers: { 'X-API-Key': apiKey },
});
const images = await res.json();
console.log('Available images:', images);

const sandbox = await vr.createSandbox({
  name: 'custom-app',
  image: 'code',
  cpu: 2,
  mem: 2048,
});
```

```python Python
import os

import httpx

from voidrun import DEFAULT_API_BASE_URL, VoidRun

api_root = DEFAULT_API_BASE_URL.rstrip("/")
key = os.environ["VR_API_KEY"]

with httpx.Client() as client:
    r = client.get(f"{api_root}/images", headers={"X-API-Key": key})
    r.raise_for_status()
    print("Available images:", r.json())

vr = VoidRun(api_key=key)
sandbox = vr.create_sandbox(name="custom-app", image="code", cpu=2, mem=2048)
```

```bash cURL
curl -sS "https://platform.void-run.com/api/images" \
  -H "X-API-Key: $VR_API_KEY"
```
</CodeGroup>

## Image caching and performance

<Note>
The first creation with a large image (especially **`docker`**) can take longer while the host caches layers. Later sandboxes on the same host typically start faster.
</Note>

| Operation | First use (typical) | Cached |
|-----------|---------------------|--------|
| Create sandbox | Longer for large images | Faster |
| Resume from sleep | N/A | Often **under ~200 ms** (platform-dependent) |
| Exec | Near real-time | Near real-time |

## Best practices

<CardGroup cols={2}>
  <Card title="✅ Do" icon="check">
    - Default to **`code`** for interpreters, Node, and Python in one box  
    - Use **`max`** when you need Debian-style compatibility  
    - Use **`docker`** only when you run Docker inside the VM; size **CPU/mem** up  
  </Card>
  <Card title="❌ Don't" icon="x">
    - Invent image names that are not in **`GET /images`** or this system list  
    - Use **`docker`** for simple script runs: **`code`** or **`max`** is lighter  
    - Recreate sandboxes when **resume** would suffice  
  </Card>
</CardGroup>

### Selection guide

| Use case | Image | Why |
|----------|-------|-----|
| AI agents, `runCode`, npm + Python | **`code`** | Fast TTI; multi-language |
| General Linux, apt-style workflows | **`max`** | Debian base |
| Build/run images, DinD | **`docker`** | Docker daemon in the VM |
| CI for mixed JS/Python | **`code`** | One image for both stacks |

### Environment variables at create

<CodeGroup>
```typescript TypeScript
import { VoidRun } from '@voidrun/sdk';

const vr = new VoidRun({ apiKey: process.env.VR_API_KEY! });

const sandbox = await vr.createSandbox({
  name: 'configured-app',
  image: 'code',
  cpu: 2,
  mem: 2048,
  envVars: {
    NODE_ENV: 'production',
    DATABASE_URL: 'postgres://localhost:5432/mydb',
  },
});

const result = await sandbox.exec({ command: 'echo $DATABASE_URL' });
console.log(result.data?.stdout);

await sandbox.remove();
```

```python Python
from voidrun import VoidRun

vr = VoidRun()
sandbox = vr.create_sandbox(
    name="configured-app",
    image="code",
    cpu=2,
    mem=2048,
    env_vars={
        "NODE_ENV": "production",
        "DATABASE_URL": "postgres://localhost:5432/mydb",
    },
)
print(sandbox.exec("echo $DATABASE_URL").data.stdout)
sandbox.remove()
```

```bash CLI
voidrun create --template code --env NODE_ENV=production --env DATABASE_URL=postgres://localhost:5432/mydb
```

```bash cURL
curl -X POST "https://platform.void-run.com/api/sandboxes" \

    "image": "code",

```
