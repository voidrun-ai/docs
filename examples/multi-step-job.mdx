# Example: Multi-step job in one sandbox

You use this pattern when **later steps depend on earlier outputs** in the same filesystem: you clone a repo, you install dependencies, you build, you run tests, you tar artifacts. You keep **one sandbox** alive for the whole pipeline so you do not re-upload layers between steps.

## What you accomplish

You run **sequential** **`exec`** calls. You set **`cwd`** so each command runs in the right tree. You pass **`env`** when a step needs extra variables. You inspect **exit codes** after expensive steps so you fail fast.

## Concepts you use

- **Shared state:** Every **`exec`** in the same sandbox sees the same disk (until you remove the VM).
- **Fail fast:** You use `set -euo pipefail` in **bash** when you wrap multiple shell commands in one **`exec`**, or you check **`exit_code`** after each SDK call.
- **Idempotency:** You design steps so retries (new sandbox) do not corrupt external systems: your job touches only paths inside the VM until you upload results out.

## Step 1: Provision workspace

<CodeGroup>
```typescript TypeScript
import { VoidRun } from '@voidrun/sdk';

const vr = new VoidRun({ apiKey: process.env.VR_API_KEY! });
const sandbox = await vr.createSandbox({
  name: 'pipeline-demo',
  mem: 4096,
  cpu: 2,
});

await sandbox.exec({ command: 'mkdir -p /work/repo && chmod 755 /work/repo' });
```

```python Python
from voidrun import VoidRun

vr = VoidRun()
sandbox = vr.create_sandbox(name="pipeline-demo", mem=4096, cpu=2)
sandbox.exec("mkdir -p /work/repo && chmod 755 /work/repo")
```
</CodeGroup>

## Step 3: Step A: prepare

<CodeGroup>
```typescript TypeScript
const prep = await sandbox.exec({
  command: 'set -euo pipefail; ls -la /work/repo > /work/repo/prep.log',
  cwd: '/work/repo',
  timeout: 60,
});
if (prep.data?.exitCode !== 0) throw new Error(prep.data?.stderr);
```

```python Python
prep = sandbox.exec(
    "set -euo pipefail; ls -la /work/repo > /work/repo/prep.log",
    cwd="/work/repo",
    timeout=60,
)
if prep.data.exit_code != 0:
    raise RuntimeError(prep.data.stderr)
```
</CodeGroup>

## Step 4: Step B: transform (depends on A)

<CodeGroup>
```typescript TypeScript
const build = await sandbox.exec({
  command: 'set -euo pipefail; cat prep.log app.txt | wc -l > /work/repo/build.count',
  cwd: '/work/repo',
  timeout: 60,
});
if (build.data?.exitCode !== 0) throw new Error(build.data?.stderr);
```

```python Python
build = sandbox.exec(

    cwd="/work/repo",
    timeout=60,
)

    raise RuntimeError(build.data.stderr)
```

</CodeGroup>

## Step 5: Step C: package artifact

<CodeGroup>
```typescript TypeScript
const pack = await sandbox.exec({
  command: 'set -euo pipefail; tar -czf /work/out.tgz -C /work/repo .',
  timeout: 120,
});
if (pack.data?.exitCode !== 0) throw new Error(pack.data?.stderr);

const artifact = await sandbox.fs.downloadFile('/work/out.tgz');
console.log('artifact bytes', artifact.length);
```

```python Python
pack = sandbox.exec(
    "set -euo pipefail; tar -czf /work/out.tgz -C /work/repo .",
    timeout=120,
)
if pack.data.exit_code != 0:
    raise RuntimeError(pack.data.stderr)

artifact = sandbox.fs.download_file("/work/out.tgz")
print("artifact bytes", len(artifact))
```
</CodeGroup>

## Step 6: Remove the sandbox

<CodeGroup>
```typescript TypeScript
await sandbox.remove();
```

```python Python
sandbox.remove()
```

</CodeGroup>

## Practices you should follow

- You merge **trivial** shell steps into one **`exec`** when that reduces round trips: balance readability against latency.
- You stream **logs** to your observability stack when steps run longer than a few minutes ([Code execution](/core-concepts/code-execution)).
- You cap **total job time** at the orchestrator level (queue worker deadline) even when each **`exec`** has its own **`timeout`**.
